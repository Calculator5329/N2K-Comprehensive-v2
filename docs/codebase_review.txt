================================================================================
1.  PROJECT AT A GLANCE
================================================================================

What it is:
- A TypeScript port of an originally-Python game-math solver for "N2K", a
  dice/equation game whose canonical equation shape is:
       d1^p1  o1  d2^p2  o2  d3^p3  =  total
  with operators evaluated STRICTLY LEFT-TO-RIGHT (no precedence).
- Ships as TWO npm workspaces in one repo:
    a) Root package "n2k-comprehensive-solver" — the Node CLI/REPL solver,
       data exporter, and the pure algorithmic core.
    b) "web/" package "@n2k/web" — a static React + MobX + Tailwind + Vite
       single-page app ("The N2K Almanac") that browses, searches, and
       visualizes the solver's bulk export, with 16 visually distinct
       editions/themes.
- Node ESM (`"type": "module"`, NodeNext resolution); Node 20+; TS 5.7
  with strict + noUncheckedIndexedAccess + exactOptionalPropertyTypes.
- Tests: Vitest, ~63 tests in 9 files at audit time.

What "comprehensive" means here:
- The solver enumerates every (dice perm × exponent triple × operator pair)
  for a triple and target. With dice 1..20 and totals 1..999 the bulk export
  produces ~370k–530k records (~30 MB) in ~6s on a modern laptop.
- The web app renders that dataset in four primary views (Lookup / Explore
  / Visualize / Compose) plus a Colophon, all client-side static, with
  themable layout, dice glyph, and equation rendering.

Why it's interesting architecturally:
- Strict three-layer separation:  cli/ -> services/ -> core/
- The pure-services layer is shared between Node (REPL/exporter) and the
  web app via a Vite path alias `@solver/* -> ../src/*` (NOT a published
  package). The web also wraps it in its own services for browser-specific
  concerns (Web Workers, fetch caches).
- The theming system has THREE layers (TS theme registry, CSS-variable
  bundles, component variants for layouts/dice/equation) and a 16-edition
  registry with no per-edition forks of feature code.

--- src/core/types.ts ---------------------------------------------------
  Operator        = 1 | 2 | 3 | 4   (ADD, SUB, MUL, DIV)
  OperatorSymbol  = "+" | "-" | "*" | "/"
  DiceTriple      = readonly [number, number, number]
  Equation        { d1, d2, d3, p1, p2, p3, o1, o2, total }
  SolverInput     { dice: DiceTriple, total: number }


--- src/core/constants.ts -----------------------------------------------
  MAX_EXPONENTS         per-base exponent caps; index by dice value.
  OP                    { ADD:1, SUB:2, MUL:3, DIV:4 }
  OPERATOR_TO_SYMBOL    map.
  SYMBOL_TO_OPERATOR    map.
  ALL_OPERATORS         [1,2,3,4]
  DICE_COMBINATIONS     38 standard tournament dice triples.
  DIFFICULTY            weighted-sum config: ~7 named terms + tenFlag,
                        upper-tail compression, ceiling clamp at 100.
  BOARD                 { rows: 6, cols: 6, size: 36 }
  DIFFICULTY_BUCKETS    [[lo,hi]] pairs for histogramming.
  FLOAT_EQ_EPSILON      1e-9


--- src/services/arithmetic.ts ------------------------------------------
  applyOperator(a, b, op): exhaustive switch on Operator. NO `default`
    branch — exhaustiveness is enforced at type level only.
  evaluateLeftToRight(n1, n2, n3, o1, o2): two-step left-to-right reduce.
  permutationsOf3(a, b, c): returns all 6 permutations.


--- src/services/parsing.ts ---------------------------------------------
  operatorToString(op), parseOperator(token).
  formatEquation(eq) -> "d1^p1 o1 d2^p2 o2 d3^p3 = total".
  parseEquation(input): split on /\s+/; expects 7 tokens; throws a
    typed SyntaxError on mismatch.


--- src/services/difficulty.ts ------------------------------------------
  difficultyBreakdown(equation): single source of truth, returns
    { equation, terms, adjustments, final, ceiling }.
  difficultyOfEquation(equation): thin wrapper -> breakdown.final.

  Heuristic shape (terms, weighted):
    distance to a free base power for the target,
    counts of zeros (^0) and ones (^1) used,
    largest sub-result + distance from target,
    smallest multiplicand penalty,
    + tenFlag adjustment for ×10 simplification,
    + upper-tail compression past a threshold,
    clamped at 100.
  basesForDice uses MAX_EXPONENTS[d] ?? 1; loop is `exp < cap`.


--- src/services/solver.ts ----------------------------------------------
  SolverOptions     { depower?: boolean = true }
  easiestSolution(input, options?): best-difficulty Equation | null.
  BulkSolution      { equation, difficulty }
  allSolutions(input, options?): every match, sorted by difficulty.
  solveForAllTargets(dice, minTotal, maxTotal, options?):
    Map<total, BulkSolution>; SINGLE enumeration over the equation
    space, scoring all targets in [min,max] in one pass — the engine
    behind the bulk export.

  Hot loop:
    for each (d1,d2,d3) in 6 dice perms:
      for each precomputed exponent triple (p1,p2,p3):
        for each (o1,o2) in 4×4 ops:
          eval left-to-right, accept within FLOAT_EQ_EPSILON, score,
          keep min (or push to all results).
  buildBasesCache: per-call Map<base, number[]> so unique dice values
  precompute their power tables exactly once. (Phase 0 perf hoist.)
  depower: 4/8/16 -> base 2; 9 -> base 3 (preserves Python semantics
  by default; export sets depower=false to keep raw dice).


--- src/services/boardAnalysis.ts ---------------------------------------
  scoreBoardForDice(dice, board): per-cell easiestSolution +
    difficultyOfEquation; impossible cells contribute 100 to the
    composite (ZeroDivisionError fix from Python). Returns
    { dice, cellDifficulties, impossibleCount,
      averagePossibleDifficulty, boardDifficulty }.
  bucketResults(results): groups by DIFFICULTY_BUCKETS using
    `>= lo && < hi`. NB: the last bucket is [80, 100) — scores
    exactly 100 (i.e. fully impossible boards) fall outside ALL
    buckets. (See bug section.)
  assertValidBoard(board): length and integer/positive-ness checks.


--- src/services/generators.ts ------------------------------------------
  generateRandomBoard(rangeOrHighest?, rng?):
    overload (number) treats as max with min=1; or { min?, max }.
  generatePatternBoard(multiples?, startingNumber?):
    accepts 1, 2, OR 3 multiples (Python parity); 3-multiple form
    auto-shifts start so cells stay non-negative.
  generateRandomDice(options?, rng?): rejects (a==b==c) re-roll.
  BoardOverride / RandomBoardSpec / PatternBoardSpec / BoardSpec.
  generateBoard(spec, rng?): dispatcher that handles per-slot pinned
    overrides while keeping unique values; with overrides it preserves
    POSITIONAL layout (so a pinned slot stays at that index).


--- src/services/exporter.ts --------------------------------------------
  enumerateUnorderedTriples(min, max): C(n+2,3) triples in
    non-decreasing order.
  exportAllSolutions(outputPath, options?): NDJSON streaming write +
    sidecar manifest (perDice array). createWriteStream + drain
    backpressure; mkdir recursive.
  Output is sorted (dice asc, total asc) for reproducibility.
  Manifest schema: generatedAt, dice/total ranges, depower,
    recordsWritten, elapsedMs, perDice[].


--- src/services/competition.ts -----------------------------------------
  Largest service. Drives the web's Compose view.
  DifficultyResolver = (dice, target) => number | null
    (caller-provided oracle; Node passes easiestSolution(...)
    .difficulty; web passes a DataStore-cached version.)
  scoreBoardCells(board, dice, resolver) -> per-cell difficulty.
  summarizeBoardDifficulty(cells) -> averagePossibleDifficulty,
    impossibleCount, boardDifficulty (avg solvable + 100×impossible).
  expectedScore(board, cells, options?):
    blends three sub-strategies (inverse-difficulty harvest,
    bottom-up time budget, easiest-first time budget); rescales by
    39.48 / timeBudget; sorts the value/difficulty arrays internally
    so it's order-invariant on the board.
  rankCandidates(board, candidates, resolver, scoreOptions?):
    sorts by expectedScore DESC.
  BalancedRollsOptions / RoundAssignment / BalancedRollsResult.
  generateBalancedRolls(board, candidates, rounds, resolver, opts?):
    1) score candidates with both expectedScore and boardDifficulty,
    2) restrict to easier-by-difficulty median half,
    3) form adjacent-rank pairs by expectedScore,
    4) sample rounds pairs uniformly without replacement,
    5) per-round assign to minimize |sumExp(P1) − sumExp(P2)|;
       difficulty delta as tie-breaker;
       has an EXACT bitmask path for ≤16 rounds and a greedy swap
       fallback (default 8 passes).
  Injectable rng default Math.random for full determinism with seed.

10. CROSS-CUTTING ARCHITECTURE OBSERVATIONS
================================================================================

Layer discipline is REAL and largely respected:
  - core/ depends on nothing in-tree.
  - services/ depends only on core/ (and node:* in exporter).
  - cli/ depends on services + core; no other ingress to console.log.
  - web/ services do fetch only.
  - web/ stores own state and orchestrate; no React imports.
  - web/ components observe stores; no fetches; no business logic.

Cross-workspace seam:
  - The web app imports the SOLVER's `src/` directly via Vite alias
    `@solver/* -> ../src/*`. This is elegant but unusual:
      + No publication step; web always has the latest solver code.
      + But: solver's tsconfig excludes tests, while web's tsconfig
        adds "../src/**/*.ts" to its include — meaning solver source
        is type-checked twice (once by root, once by web/tsc -b).
      + The root tsconfig's `module: NodeNext` produces ESM output for
        the solver, while web bundles via Vite/esbuild — the source
        files are written with `.js` import suffixes (Node ESM
        convention), which Vite/Esbuild also accept since aliasing
        resolves before file lookup. This mostly works but is fragile.

RNG & determinism:
  - All random surfaces accept an injectable rng: () => number.
  - Generators tests use a seededRng helper; competition tests do too.
  - CompositionStore builds a mulberry32 from optional seed text via
    FNV-1a; this is reproducible end-to-end.

Performance philosophy:
  - Brute-force solver, intentionally. MAX_EXPONENTS keeps it tractable.
  - Hot path optimizations: per-base power tables (precomputeBases),
    single-pass solveForAllTargets, NDJSON streaming export with
    backpressure.
  - Web app: lazy per-dice JSON chunks (~3-5 KB) with in-flight
    dedupe; Web Worker for full-equation lists; no React Suspense.

Theming system:
  - Three layers: TS theme registry; CSS-variable bundles (one per
    [data-theme]); component variants (DiceGlyph, Equation, layouts).
  - 16 editions, no per-edition forks of feature code. Adding a 17th
    is a four-step recipe (themes.ts entry + CSS bundle + index.html
    whitelist + nav.ts colophon).
  - Colors are space-separated RGB triplets so Tailwind's
    `<alpha-value>` modifier (`bg-paper-100/60`) keeps working.

Hash + storage:
  - urlHashState envelope is shared between Lookup, Explore (saved
    views), and the Compose plan (planned in Phase 5).
  - localStorage keys: n2k.theme, n2k.favorites.v1,
    n2k.explore.views.v1.

Accessibility:
  - Keyboard nav implemented for the Lookup neighborhood strip and
    page-level shortcuts.
  - aria-label, aria-pressed (FavoriteToggle), aria-current
    (NeighborhoodStrip), aria-live (AllEquationsList loading),
    radiogroup/radio (ThemeSelector).
  - Likely incomplete on charts (heatmap is a giant grid of buttons or
    DOM cells; would benefit from a screen-reader summary table).

Testing posture:
  - Strong unit coverage for arithmetic, parsing, difficulty, solver
    (with caveats), generators, board analysis, competition.
  - Integration coverage on exporter via tmp dir.
  - One live solver integration in competition.
  - NO tests for prepare-web-data, web stores, or web components.

12. PERFORMANCE HOTSPOTS
================================================================================

Solver core (src/services/solver.ts):
  - Dominant cost is the inner triple loop (perms × exps × ops).
    Already optimized with per-base power tables.
  - difficultyOfEquation called per accepted candidate. Difficulty
    walks bases again — could be hoisted further to share the same
    base tables.
  - solveForAllTargets is a major win over per-target easiestSolution.

Board analysis (src/services/boardAnalysis.ts):
  - scoreBoardForDice does 36 separate easiestSolution calls per dice
    triple. Per `Find board difficulty` (#8), this is 38 × 36 = 1,368
    calls. Replacing with solveForAllTargets(dice, 1, max(board))
    once, then looking each board cell up in the resulting Map, would
    drastically reduce work (already noted in roadmap "Longer term").

Competition (src/services/competition.ts):
  - rankCandidates over the Extensive 1,540-triple pool calls
    expectedScore once per triple. expectedScore allocates and sorts
    arrays of length 36 each call. For Extensive pools with many
    boards this adds up; some array reuse would help if it ever
    becomes a UX bottleneck.

Web app:
  - Visualize heatmap: 999 DOM cells per render. With theme switches
    + scroll, this is fine, but a Canvas/SVG redraw would be cheaper
    if charts grow.
  - AllEquationsList: solver runs in worker, so main thread is okay.
    The serialized result can be large; consider streaming chunks.
  - CSS file size: ~2.1k lines. Both Tailwind-purged components plus
    raw theme bundles add up. Per-theme CSS code splitting could reduce
    initial paint cost.

13. RISK REGISTER / TECHNICAL DEBT
================================================================================

High-leverage debts:
  D1. Web app has zero automated tests. Stores like ExploreStore (~580
      LOC of filter+sort+saved-views) and CompositionStore (~290 LOC

      Vitest to the web workspace would catch regressions cheaply.

  D3. The cross-workspace alias `@solver/*` is unconventional. If you
      ever want to publish the solver as a package, this seam will
      need to evolve into a proper workspace dep (or a published npm
      package), and then the web tsconfig include that pulls in
      `../src/**/*.ts` becomes redundant/misleading.
  D4. globals.css is monolithic and scaling linearly with editions.
      A theme-per-file split would cap blast radius and make code
      review of new editions cleaner.
  D5. The "Advanced Mode" plan (docs/current_task.md) is large and
      sprawling. It introduces a custom binary file format, new
      arity-N types, a separate worker pool, and a hidden Æther
      edition. Every one of these is worth doing only if the core
      product needs the data; the plan doesn't say WHY (no user-facing
      need is documented).

Medium debts:
  D6. ESLint is declared but not configured. CI is presumably absent
      (no .github/, .gitlab-ci, etc. seen in audit).
  D7. Repo root contains ~80 PNG screenshots (~10+ MB total). They
      double as reference shots and as "did we break it" baselines,
      but they're not in a git-friendly subfolder and they're not in
      .gitignore. Risk: bloat over time.
  D8. competition.ts is the largest single file at ~580 LOC and packs
      many responsibilities (scoring, ranking, sampling, balancing,
      bitmask optimizer). Splitting into competition/scoring.ts +
      competition/balancing.ts would make it more testable.

Low debts:
  D9. Stale README sections (three editions vs sixteen).
  D10. Confusing dataset drift between manifest and shipped JSON.
  D11. Dead/stale comments and assertions noted in §11.

       in vitest config) and post coverage on PRs.

LARGE (multi-day):
  L1.  Browser-side bulk export. The data pipeline currently requires
       Node + tsx. Move the export into a Web Worker pool (already
       hinted in the Advanced Mode draft) and let the web app
       generate its own dataset. Side benefit: kills the cross-
       workspace alias's fragility because the web no longer needs
       Node-side scripts.
  L2.  Better solver algorithm. The current brute-force solver is
       O(perms × exp^3 × ops^2). For each (dice triple) you can:
         a) Precompute all reachable values from a single die under
            the chosen exp range.
         b) For each pair of dice, build a sorted Map of all
            (op-applied) intermediate values (meet-in-the-middle).
         c) For each candidate target, look up complementary
            intermediate values in the third die's reachable set.
       This drops the per-target check from O(d^3) to O(d^2 log d)
       and makes Advanced Mode (43 dice × 5,000 targets × arity 4/5)
       tractable without 8-worker pools.
  L3.  Parquet output. Add an `--out-format=parquet` mode for the
       export. NDJSON is convenient; Parquet is cheaper to query
       and load into DuckDB/Polars and would shrink the dataset
       roughly 5-10×. Requires accepting a single dependency
       (`parquetjs` or similar).
  L4.  Theme system as a plugin tree. Today, adding an edition
       touches four files. With a small plugin loader you could
       co-locate everything for one edition (theme record + CSS
       bundle + optional layout + optional dice variant + colophon)
       in a single `editions/<id>/` folder, registered via a
       `import.meta.glob` discovery. The system already scales
       fine; this is a maintainability nicety once you cross ~25
       editions or want third parties to drop in editions.
  L5.  Reconsider Advanced Mode scope. The draft in
       docs/current_task.md is a major lift (custom binary format,
       new arity types, separate worker pool, secret edition). The
       only documented motivation is "vastly enlarges the playable
       space." Before building it, write down WHO benefits and HOW
       the secret data shows up in their workflow. If the core
       audience is players of the original 3-dice game, Advanced is
       optional flavor — and a much smaller solver-only export to
       NDJSON for a power user might be 90% of the value at 10% of
       the cost.
  L6.  Replace the cross-workspace alias `@solver/*` with a real
       internal package. Move `src/` into `packages/solver/src/`,
       give it its own `package.json`, and have web depend on
       `"@n2k/solver": "*"`. This unwinds the dual typecheck (root
       tsc + web tsc -b each compile the solver), formalizes the
       contract, and makes the future "publish solver as a package"
       option a one-line change.

ARCHITECTURAL OPTIONS (pick ONE if you want a sweeping change):
  A1. "Solidify and ship". Treat the current product as feature-
      complete. Land L6 (workspace), L1 (browser-side export), and
      M8 (CI). Skip Advanced Mode entirely. Result: small,
      maintainable repo that any contributor can own.
  A2. "Lean into the almanac". Keep the editions, kill the CLI as a
      product surface (it stays as a dev tool), and double down on
      the web app: more visualizations, embeddable widgets, sharable
      compositions, deep-link competition plans (Phase 5 of roadmap).
      The CLI then exists just to refresh the dataset.
  A3. "Become the algorithm playground". Generalize the solver to
      arbitrary arity (≥1) and treat the heuristic as user-editable
      (publish weights as JSON, allow users to A/B their own
      heuristic). Calibrate with a labeled corpus. Result: useful
      to other recreational-math researchers, not just N2K players.
